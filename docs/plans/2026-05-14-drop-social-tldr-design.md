# Design ÔÇö Drop the `tldr` social-post field

Date: 2026-05-14
Owner: Aman
Linear: (n/a ÔÇö internal)

## Problem

PR #136 (merged) added two new LLM-generated digest-level fields, `hook` and `tldr`, for the LinkedIn long-form post and X thread:

```
<hook>

TLDR: <tldr>

1) <story 1 title>
   <story 1 summary>

2) <story 2 title>
   <story 2 summary>
ÔÇŽ
```

End-to-end testing showed `tldr` doesn't earn its place. The LLM output reads like marketing copy ("Anthropic launched Claude for Small Business today, shipping 15 ready-to-run agentic workflows inside QuickBooks, PayPal, HubSpot, Canva, and DocuSignÔÇŽ") and adds clutter without adding information ÔÇö the numbered story list immediately below already conveys the same content in scannable form. On a 12-story digest the `TLDR:` paragraph pushes the post deeper into "click to read more" territory before the reader sees any story.

`hook` stays ÔÇö it's a strong one-line opener that genuinely changes how the post lands.

## Goals

1. Drop the `tldr` field from the digest-level LLM output (Zod schema + prompt).
2. Drop the `tldr` column from `run_archives` (migration 0017).
3. Strip `tldr` from the composer signature, both notifiers, the API read path, and the web type.
4. LinkedIn post becomes `hook Ôćĺ numbered stories Ôćĺ archive link`. X thread opens with `hook` only.

## Non-goals

- Do **not** touch `hook` ÔÇö it ships as-is.
- Do **not** change `digest_headline` / `digest_summary` (still drive the archive UI and listing).
- Do **not** change the LinkedIn-over-3000-char failure path (still surfaces in `social_metadata.linkedinError`).

## Architecture

### LLM schema

`packages/pipeline/src/processors/rank.ts`:

```ts
// before
const digestSchema = z.object({
  headline: z.string(),
  summary: z.string(),
  hook: z.string(),
  tldr: z.string(),
});

// after
const digestSchema = z.object({
  headline: z.string(),
  summary: z.string(),
  hook: z.string(),
});
```

`RankResult` loses `tldr: string`. `rank-prompts.ts` loses the `digest.tldr:` block; the surrounding "Also return a social-post hookÔÇŽ" preamble is trimmed from "two social-post fields" to "a social-post hook."

### Storage

Migration `0017_drop_tldr.sql`:

```sql
ALTER TABLE "run_archives" DROP COLUMN "tldr";
```

Generated by `drizzle-kit generate` from the schema edit. Zero-downtime ÔÇö the column is unused by the API after this PR lands, and the migration runs after the new code is deployed.

### Write path

`packages/pipeline/src/workers/run-process.ts` drops the `tldr` extraction and the `tldr` field from the `archivesRepo.upsert(...)` call. `pickArchiveDigest` was never touched by tldr ÔÇö no change there.

### Repository

Both `PipelineRunArchiveRow` and `RunArchiveUpsertInput` lose `tldr`. The insert payload and the `onConflictDoUpdate` set lose the `tldr` line. All three projections in `packages/api/src/repositories/run-archives.ts` lose `tldr: runArchives.tldr`.

### Read path

`GET /api/archives/:runId` no longer exposes `tldr` on the response. The web `RunStateResponse` type loses the optional `tldr` field.

### Composer

`composePosts` signature changes:

```ts
// before
export interface ComposeInput {
  hook: string | null;
  tldr: string | null;
  stories: RankedStory[];
  archiveUrl: string;
}

// after
export interface ComposeInput {
  hook: string | null;
  stories: RankedStory[];
  archiveUrl: string;
}
```

`buildLinkedin` no longer prepends a `TLDR: <tldr>` line. `buildTwitterOpener` is gone ÔÇö tweet 1 is just the hook (which the prompt bounds to ÔëĄ140 chars, well under 280).

### Notifiers

Both `linkedin/notifier.ts` and `twitter/notifier.ts`:
- `ArchiveLike` interface loses `tldr: string | null`.
- The `composePosts({ ÔÇŽ })` call loses the `tldr: archive.tldr` line.

## External Dependencies & Fallback Chain

None. This is a pure deletion ÔÇö no new deps, no version changes, nothing to probe.

## Risks

- **Migration ordering.** PR #136's `0016_add_hook_tldr.sql` is already in main and applied in prod. `0017_drop_tldr.sql` is the forward fix ÔÇö it depends on `tldr` existing, which it does in any DB that ran 0016. The local dev DB on my machine has had `tldr` manually removed during the previous session; that drift is local-only and doesn't affect prod or CI. No production data is lost because `tldr` was never user-edited or downstream-consumed beyond the social-post composer.
- **In-flight runs.** A run started before the deploy will write a row with `tldr` populated by the LLM; the new migration drops the column. Drizzle's insert no longer includes `tldr`, so there's no schema mismatch. The two events are independent.
- **Tests.** Every test fixture that constructed a `PipelineRunArchiveRow` or stubbed the LLM `digest` object had to lose the `tldr` field. Done.
