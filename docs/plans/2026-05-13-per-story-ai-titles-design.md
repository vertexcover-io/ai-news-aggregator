# Per-Story AI Titles — Design

**Date:** 2026-05-13
**Status:** Approved
**Linear:** _(none — internal task)_

## Problem

Today the newsletter pipeline generates a digest-level `{headline, summary}` and per-story recap content (`summary`, `bullets`, `bottomLine`) via Claude Haiku, but the **title shown on each story card is the original scraped source title** (`raw_items.title`). Source titles are inconsistent: some are HN-style understatements, some are SEO blog headlines, some are tweets, some are GitHub repo names. The result is a visually and tonally uneven digest.

We want a single, consistent, newsletter-quality title per story — generated by the LLM and editable in admin review.

## Goal

Generate a short, neutral-newswire title (4–7 words, ~50 chars) for every story in the daily digest. The AI title replaces the scraped source title in all UI surfaces (public archive listing, single archive page, admin review). Operators can edit the title inline in admin review. Archives created before this change fall back to the source title.

## Non-goals

- Backfilling AI titles for past archives.
- Hard runtime length enforcement (we trust the LLM + clear prompt instructions; review UI handles outliers).
- Multi-language support.
- A/B testing of title styles.
- Changing the digest-level headline (already exists, untouched).

## Design Decisions

### Title style: tight, neutral newswire

- **Length:** 4–7 words, ~50 characters.
- **Tone:** Neutral newswire. Subject–verb–object, sentence case, names the actor and the action. No clickbait, no questions, no colons-as-title-tropes, no editorial framing words ("quietly", "finally", "doubles down").
- **Examples to embed in the prompt:**
  - `OpenAI ships GPT-5 with native tool use`
  - `Anthropic raises $5B at $60B valuation`
  - `Meta open-sources Llama 4 weights`
  - `Google’s Veo 3 lands on Vertex AI`

### Generation point: inside the recap step

`recapContentSchema` (packages/pipeline/src/processors/recap.ts) already produces `{summary, bullets, bottomLine}` per shortlisted story via Claude Haiku. We add `title` as a fourth field on the same structured output.

- **Zero new model calls.** Recap already runs once per shortlisted story.
- The title sees the full article content the recap sees — best context for an accurate, specific headline.

### UI source resolution: three-tier precedence in hydration

`hydrateRankedItems` (packages/api/src/services/rank-hydration.ts) resolves the displayed title in this order:

```
ref.title  >  rawRecap?.title  >  row.title
```

1. **`ref.title`** — operator override saved in `run_archives.ranked_items` (`RankedItemRef.title`). Wins if present.
2. **`rawRecap?.title`** — AI-generated title from `raw_items.metadata.recap`.
3. **`row.title`** — original source title from `raw_items.title`. The fallback for old archives.

The frontend continues reading `item.title` — hydration takes care of resolution. No shape change at the API boundary.

### Backwards compatibility: no migration

- `recap.title` is **required** in the Zod schema going forward → all new runs produce it.
- Old `raw_items.metadata.recap` rows have no `title` field → `rawRecap?.title` is `undefined` → hydration falls back to `row.title`.
- Old `RankedItemRef` entries in `run_archives.ranked_items` have no `title` field → `ref.title` is `undefined` → hydration falls back to AI title or source title.
- No DB migration. No data backfill.

### Add-post-by-URL: automatic

`add-post-helper.ts` (`POST /api/admin/archives/:runId/add-post`) already calls the recap function when an operator adds a story manually. Once `recapContentSchema` includes `title`, this path automatically generates an AI title — no extra code.

## Architecture (single phase, ~8 files)

### 1. Shared types
- **`packages/shared/src/types/index.ts`** — `RecapContent` interface: add required `title: string`.
- **`packages/shared/src/types/run.ts`** — `RankedItemRef` interface: add optional `title?: string` (operator override slot).
- **`RankedItem`** already has `title: string`. We override it in hydration; no shape change.

### 2. Pipeline — recap generation
- **`packages/pipeline/src/processors/recap.ts`**
  - `recapContentSchema`: add `title: z.string().min(1)`.
  - `RECAP_SYSTEM_PROMPT`: prepend a `TITLE:` section before the existing SUMMARY/BULLETS/BOTTOM LINE sections:
    > **TITLE:** A 4-to-7-word neutral newswire headline summarizing the story. Sentence case. Names the actor and the action. No clickbait, no questions, no colons-as-title-tropes, no editorial framing words. Examples: "OpenAI ships GPT-5 with native tool use", "Anthropic raises $5B at $60B valuation", "Meta open-sources Llama 4 weights".

### 3. API — hydration
- **`packages/api/src/services/rank-hydration.ts`** — `hydrateRankedItems`: resolve the displayed title using the three-tier precedence above. The function still returns `RankedItem` with the resolved `title` in the existing field — the frontend shape is unchanged.

### 4. API — review patch
- **`packages/api/src/lib/validate.ts`** — `patchArchiveSchema`: add optional `title: z.string().min(1).max(160).optional()` on each item.
- **`packages/api/src/services/review.ts`** — `patchArchive`: if `i.title !== undefined`, set `ref.title = i.title` on the `RankedItemRef`.

### 5. Web — admin review UI
- **`packages/web/src/components/review/ReviewCard.tsx`**
  - Extend `onUpdateField` field-name union to include `"title"`.
  - Add an `EditableField` at the top of the card for the title (same pattern as summary/bullets/bottomLine).
- **`packages/web/src/pages/AdminReviewPage.tsx`** (or wherever the PATCH body is assembled) — include `title` in the diff sent to PATCH when the operator has edited it.

### 6. Display surfaces — no changes needed
- Public archive listing (`/`), single archive page (`/archive/:runId`), admin review (`/admin/review/:runId`) all already read `item.title`. Hydration now resolves to the AI title (or override, or source fallback) transparently.

## Data flow

```
raw_items row
  ├─ title:            "Show HN: a tiny vector DB I built last weekend"   ← scraped
  └─ metadata.recap:
       ├─ title:       "Solo dev releases lightweight vector database"     ← AI
       ├─ summary:     ...
       ├─ bullets:     ...
       └─ bottomLine:  ...

run_archives.ranked_items[i]  (RankedItemRef)
  ├─ rawItemId:        42
  ├─ title?:           "Indie dev ships sub-1MB vector DB"   ← optional operator override
  └─ ...

hydrateRankedItems returns RankedItem:
  title = ref.title ?? rawRecap?.title ?? row.title
        = "Indie dev ships sub-1MB vector DB"
```

## Testing

| Layer | Tests |
|---|---|
| Pipeline | recap step: schema includes `title` (required, min length 1); prompt mentions 4–7 words and newswire tone |
| API | `hydrateRankedItems`: precedence `ref.title > rawRecap.title > row.title`; each tier individually missing falls through correctly |
| API | `patchArchive`: when request body has `title`, the stored `RankedItemRef` gains `title` |
| API | `patchArchiveSchema`: accepts 1–160 char titles; rejects empty/>160 |
| Web | `ReviewCard`: renders an editable title field; editing emits `onUpdateField(id, "title", value)` |

## Verification scenarios

- **VS-1:** A fresh run produces stories where `recap.title` is a 4–7 word neutral headline. Verify via DB inspection of `raw_items.metadata.recap.title` after a "Run Now".
- **VS-2:** The archive page for that run displays the AI title (not the original `raw_items.title`).
- **VS-3:** In admin review, the operator changes a title and saves; reloading the public archive shows the operator's title.
- **VS-4:** Loading an archive created before this change still renders titles (falling back to source titles), with no errors.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| LLM occasionally produces a title >7 words or off-tone | Operator edits in review. Length is a soft target in the prompt, not enforced at runtime. |
| Existing `raw_items` rows have recap with no title | Optional chaining + fallback to `row.title`. Verified by unit test. |
| `RankedItemRef.title` JSONB column collisions | None — it's a new optional key on an existing JSONB structure. |
