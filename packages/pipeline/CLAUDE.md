# @newsletter/pipeline

BullMQ workers that collect, process, and prepare newsletter items.

## Responsibilities
- Collectors fetch from external sources (HN, Reddit, etc.) and upsert into `raw_items`
- Processors transform/dedup/rank items as pure functions called from workers
- The `run-process` worker is the single BullMQ job per run â€” it runs all requested collectors concurrently in-process, then dedups and ranks
- Services own cross-cutting state (Redis run-state, candidate loading) shared by workers
- Repository modules handle DB access (via `@newsletter/shared` schema)

## Layout
- `src/collectors/` â€” one file per source (`hn.ts`, `reddit.ts`, `web.ts`); each exports both the batch collector (`collectHn`, `collectReddit`, `collectWeb`) called by the run-process worker AND the single-post fetcher (`fetchHnPost`, `fetchRedditPost`, `fetchWebPost`) used by the add-post flow. `health/` sub-package exports 5 per-collector health check strategies (`checkHnHealth`, `checkRedditHealth`, `checkTwitterHealth`, `checkWebSearchHealth`, `checkBlogHealth`) and `runAllHealthChecks` orchestrator called by the `health-check` worker. The `twitter/index.ts` collector also exports a single-tweet fetcher `fetchTwitterPost(url, deps)` + `parseTweetIdFromUrl(url)` for the add-post flow â€” these are wired by `services/add-post-helper.ts`'s `dispatchFetch`. Twitter URL detection (`x.com` / `twitter.com` / `mobile.*` + `/status/<id>`) is checked **before** HN/Reddit/web in `detectAddPostSourceType` so a Twitter status URL never falls through to the generic web crawler. `web.ts` also exports LLM extraction helpers (`discoverPostUrls`, `extractPostFields`, `validateDiscoveredUrls`, `sortPostsByPublishedAtDesc`, `applySinceDays`, `parseDateOrNull`) and `buildRawItem`. The two web-collector LLM calls (`discoverPostUrls` + `extractPostFields`) run on `deepseek-chat` (= V4 Flash) via `@ai-sdk/deepseek` (`WEB_COLLECTOR_MODEL_ID` const; `resolveDefaultModel` builds the model via `createDeepSeek({ apiKey: process.env.DEEPSEEK_API_KEY })`) â€” distinct from the Anthropic-based shortlist/rerank/recap stages. Date handling: `web-date.ts` exports `resolvePublishedDate(raw, referenceDate)` which resolves relative ("4 hours ago") and natural-language date strings via `chrono-node`, falling back to `Date.parse`, returning `null` on failure. `buildRawItem` prefers the structured `publishedAt` extracted from DOM signals (passed as 4th argument from the detail-fetch `CrawlResult`) over the LLM-returned `published_at` string; both `sortPostsByPublishedAtDesc` and `applySinceDays` route LLM date strings through `resolvePublishedDate` so relative listing dates sort and filter correctly. `fetchWebPost` sets `publishedAt` from the structured extractor rather than hardcoding `null`. `validateDiscoveredUrls(posts, listingMarkdown, listingUrl)` takes the source's `listingUrl` and **resolves relative discovered hrefs to absolute** against it (the discovery LLM commonly returns relative links as they appear in the page markdown), dropping empty/fragment-only/non-`http(s)` URLs â€” without this the relative URL reaches Crawlee's `addRequests`, which validates the whole batch atomically and aborts the entire `blog` crawl on a single bad URL. `services/web-crawler.ts::runWebCrawl` keeps an independent backstop (`isCrawlableUrl`) that filters non-`http(s)` URLs out of the batch and records them as `{ ok: false, error: "invalid-url" }`, so no future caller can re-trigger the whole-batch abort. The `twitter/` subdirectory uses `rettiwt-api` via the `RettiwtClient` adapter (`clients/rettiwt.ts`); `denormalize()` unwraps retweets first (`inner = t.retweetedTweet ?? t`) and then extracts any `inner.quoted` into `NormalizedTweet.quotedTweet` so the mapper (`map.ts`) can append a `Quoting @handle: â€¦` block to `RawItem.content` and persist the structured quoted-tweet under `metadata.quotedTweet` â€” surfacing quoted-tweet text to ranker/recap LLMs. Link enrichment continues to read URLs from the outer (or retweet-unwrapped) tweet only; quoted URLs are stored but not fetched. The `web-search/` subdirectory implements the open-web search collector: `web-search/index.ts` exports `collectWebSearch(deps, config)` which runs each query through a pluggable `WebSearchProvider` interface; `web-search/providers/types.ts` defines the interface + `WebSearchResult` type; `web-search/providers/tavily.ts` implements `TavilyProvider` (uses `@tavily/core`, `sourceType: "web_search"`, `externalId: "tavily:<sha256(url)>"`); `web-search/providers/index.ts` exports the `createWebSearchProvider(name, env)` factory (requires `TAVILY_API_KEY` from env). Adding a future provider = one new file implementing `WebSearchProvider` + one line in the factory.
- `src/processors/` â€” pure stage functions (`dedup.ts`, `shortlist.ts`, `rank.ts`); `shortlist.ts` is the stage-1 recency-decay shortlist, and `rank.ts` is the stage-2 Claude Haiku reranker using Vercel AI SDK `generateObject` with a 3-axis prompt (Novelty, Signal-vs-hype, Actionability), with prompts inlined as TS consts in `rank-prompts.ts`; `recap.ts` is a standalone `generateRecap()` helper used by the add-post flow to generate recap content for a single added item.
- `src/queues/` â€” BullMQ `Queue` definitions (`processing.ts` is the only queue the API enqueues to; `collection.ts` is kept in place for rollback and no longer receives new jobs)
- `src/workers/` â€” a single dispatching `Worker` in `processing.ts` that routes jobs by `job.name` to processor functions; `run-process.ts` exports `handleRunProcessJob()` (collects via `Promise.allSettled`, dedups, ranks). `health-check.ts` exports `handleHealthCheckJob()` which runs per-collector health strategies in parallel, sends Slack notification on failure (debounced for scheduled checks). During the collect stage it wraps the injected `rawItemsRepo` so every item written by a collector is stamped with the current `runId` (`upsertItems(items.map((i) => ({ ...i, runId })))`) â€” this `raw_items.run_id` pointer is what the calendar eval uses to attribute the candidate pool to a specific run. `daily-run.ts` exports `handleDailyRunJob()` (loads saved settings and calls `startRun()`). `collection.ts` is legacy and left in place for rollback.
- `src/index.ts` â€” process entrypoint; creates a shared Redis connection used by both the processing worker and the `RunStateService`. The `processingWorker.on("failed")` handler calls `runState.setStage(runId, "failed", "failed")` for `run-process` jobs so stalled or otherwise externally-failed jobs propagate their state to Redis (and the UI stops polling as `"running"` forever). The `processingWorker.on("stalled")` handler logs a warning â€” BullMQ will retry automatically; state is only updated when stall exceeds `maxStalledCount` and BullMQ fires `failed`. Exports `getRunIdFromJobData` for extracting `runId` from raw BullMQ job data.
- `src/services/` â€” `run-state.ts` (Redis-backed per-run status read/write), `candidate-loader.ts` (loads `raw_items` rows for ranking), `recency.ts` (half-life decay helper), `web-crawler.ts` (Crawlee `AdaptivePlaywrightCrawler` wrapper â€” `runWebCrawl(jobs, opts)`), `web-fetch/` (`types.ts`, `index.ts`, `convert.ts`, `fetch-adaptive.ts`, `fetch-static.ts`, `fetch-browser.ts`, `published-date.ts`, `proxy.ts` â€” Crawlee+Readability+Turndown HTMLâ†’markdown pipeline; `published-date.ts` exports `extractPublishedAt(doc)` which reads the original DOM (before Readability mutation) for a publish date in precedence order: JSON-LD `datePublished` â†’ `<meta>` published-time variants â†’ `<time datetime>`; `ConvertResult` carries `publishedAt: Date | null` threaded through all three fetch functions. **Proxy support (`proxy.ts`):** `resolveWebProxyUrl(env?)` returns the trimmed `WEB_HTTP_PROXY` value for a valid `http(s)` URL or `null` (fail-open â€” unset/empty/whitespace/malformed/non-http all resolve to direct egress; malformed values log a single non-secret `warn` carrying only `{event,reason}`, never the value). The resolved URL is wired into the web collector's three outbound-HTTP seams: `fetchStatic` attaches an undici `ProxyAgent` as the per-request `dispatcher` **only on the default `globalThis.fetch` path** (an injected `fetchFn` owns its own transport and is never proxy-wrapped â€” REQ-006); `fetchBrowser` launches chromium with `proxy:{server,username,password}` parsed from the URL (`parseProxyForPlaywright`, URL-decoded credentials); and `runWebCrawl` passes `new ProxyConfiguration({proxyUrls:[url]})` to `AdaptivePlaywrightCrawler`, covering both the static and adaptive-browser sub-paths. The proxy URL is a secret â€” it is never logged in any seam and `crawler.stats` emits no proxy field. `undici@7.24.7` is pinned explicitly in `packages/pipeline/package.json` (it was a phantom-transitive dep â€” present in the pnpm store via crawlee/playwright but not importable under pnpm's strict layout until declared)), `link-enrichment/` (`types.ts`, `url-classifier.ts`, `cache.ts`, `fetcher.ts`, `index.ts` â€” `enrichRawItems(items, ctx)` enriches Reddit/HN/Twitter items' external URLs in place via `fetchAdaptive`; classifier skips self-posts, same-platform links, non-HTML media, and cache hits; one shared `EnrichmentContext` per run carries the URL cache, counters, run-level AbortSignal, and logger), and `add-post-helper.ts` (`hydrateAddedPost()` orchestrates single-item fetch + recap; `dispatchFetch()` forwards both `signal` and `fetchFn` to the appropriate single-post collector)
- `src/repositories/` â€” Drizzle wrappers like `createRawItemsRepo(db)`; `upsertItems` persists the per-row `runId` and updates it in the `onConflictDoUpdate` set clause (so a re-collected item's `run_id` moves forward to the latest run). The run-archives repo exposes `setCostBreakdown(runId, breakdown)` which UPDATEs `run_archives.cost_breakdown` (callers must guarantee the archive row exists first â€” failure paths in `workers/run-process.ts` insert a partial archive row before persisting cost). `eval-exports.ts` (`createEvalExportsRepo`) powers `/admin/eval` Calendar mode: `getCompletedRunDetail(runId)` and `listCompletedRunsByDate(date, tz)` build a run's candidate pool via the private `loadDedupedPool(db, archive)` â€” load `raw_items WHERE run_id = archive.id`, fall back to the `collectedAt` window only when no rows carry the `run_id` (pre-migration archives), then apply `dedupCandidates` and expose the surviving items as `sourcePool`. `itemCount` is `sourcePool.length` on both code paths, so the calendar list row and the loaded detail report the same deduped pool size; `buildPreviousRanking` still renders rows from the stored `RankedItemRef` when an id is absent from the deduped pool.
- `src/services/credential-resolver.ts` â€” `resolveLinkedInCredentials()` / `resolveTwitterOAuth1Credentials()` implement the DB-first / env-fallback resolver used to construct the LinkedIn and Twitter notifiers in `workers/processing.ts::buildDefaultPublishDeps`. Resolvers are invoked **per pipeline job**, not at worker startup, so admin saves at `/admin/settings` take effect on the next job without a worker restart. The DB read goes through `repositories/social-credentials.ts` (uses `getCredentialCipher()` from `@newsletter/shared/services/credential-cipher`); env fallback consumes `LINKEDIN_*` / `TWITTER_*` keys. If neither source yields complete credentials, the resolver returns `null` and the notifier is skipped.
- `src/repositories/social-tokens.ts` â€” `createSocialTokensRepo(db, cipher)` wraps the `social_tokens` table for encrypted OAuth token storage. Exposes `saveToken({ platform, accessToken, refreshToken, expiresAt, metadata })` (AES-256-GCM encrypt via `cipher.encrypt()`; upserts on `platform`), `getToken(platform)` (decrypts and returns `{ accessToken, refreshToken, expiresAt, metadata }` or `null`), and `withTokenLock(platform, cb)` (FOR UPDATE row-level lock for token-refresh racing in the LinkedIn notifier). All tokens are encrypted at rest â€” a raw DB read of `social_tokens.encrypted_fields` shows ciphertext only. The cipher is wired by the caller (`getCredentialCipher()` from `@newsletter/shared/services/credential-cipher`) so the pipeline and API share the same encryption key derived from `SESSION_SECRET`.
- `src/services/run-logger.ts` â€” `createRunLogger(runId, { repo, logger })` returns a level-method object (`info`/`warn`/`error`/`debug`). Each call appends exactly one `run_logs` row via `repositories/run-logs.ts::createRunLogRepo(db).append` (a pure INSERT, no row-existence precondition) AND emits the existing Pino line; reserved keys (`stage`/`source`/`event`) route to columns, the remainder to the `context` JSONB (null when empty). The insert is **best-effort** â€” a throwing insert is caught, logged to stdout as `run_log.write_failed`, and swallowed so a run never fails because logging failed (REQ-016). `workers/run-process.ts` threads milestone events through this logger: `run.started`, paired `stage.start`/`stage.end` (`context.durationMs` on end), `source.completed`/`source.failed` per source, three `stage.result` rows (dedup/shortlist/rank `inputCount`/`outputCount`), one `enrichment.summary` (EnrichmentTelemetry snapshot), and a `level="error"` `run.failed` row carrying `message`/`stage`/`fatal`/`context.stack` on a fatal abort. At finalize the worker computes the `RunFunnel` `{collected, deduped, shortlisted, ranked}` and passes it into `archiveRepo.upsert` â€” written inside the existing finalize upsert (additive `runFunnel` field on insert + onConflictDoUpdate; no standalone UPDATE writer; partial funnel with later stages null on a mid-stage failure). The default deps factory wires `createRunLogRepo(db)` as `RunProcessDeps.runLogRepo`.
- `src/services/cost-tracker.ts` â€” `createCostTracker(runId)` returns `{ record, snapshot, merge }`. Wired into the four LLM call sites (`collectors/web.ts::discoverPostUrls` and `extractPostFields` â€” both on `deepseek-chat` via `@ai-sdk/deepseek`; `processors/rank.ts::rerank` and `processors/recap.ts::generateRecap` â€” on Anthropic Claude Haiku) via an optional `reportUsage`/tracker parameter; per-stage usage is priced through the provider-aware `extractUsage(modelId, ...)` dispatcher from `@newsletter/shared` (deepseek- ids â†’ DeepSeek extractor, gemini- ids â†’ standard-field extractor, all others â†’ the unchanged Anthropic extractor), so the run-process worker finalises and persists `tracker.snapshot()` to `run_archives.cost_breakdown` on success / failure / cancellation, and `add-post-helper.ts` uses `tracker.merge(existing).snapshot()` to add-post-merge into an existing breakdown.
- `src/social/` â€” auto-post integrations that fire from the `newsletter-send` worker after subscriber email + Slack send-summary. `linkedin/` and `twitter/` each contain `oauth.ts` (token refresh â€” reads/writes `social_tokens` via `withTokenLock` to prevent parallel-refresh races; the access token stored in `social_tokens` is the authoritative source after the admin OAuth connect flow), `api-client.ts` (low-level platform HTTP â€” LinkedIn's `createComment` calls `socialActions/{postUrn}/comments`; Twitter's `createPost` accepts `replyToTweetId` for threaded replies), and `notifier.ts` (the `notifyArchiveReady({ runId })` entrypoint that composes message text, posts the head post, then posts the archive URL as a comment (LinkedIn) or reply tweet (Twitter), marks the run with `linkedin_posted_at` / `twitter_posted_at` + `social_metadata`, and returns a `posted | skipped | already_posted | failed` result; a comment/reply failure logs a warning but never demotes a successful head post to a failure). `compose.ts` builds the platform-agnostic message body â€” note that the archive URL is **never** embedded in either the LinkedIn body or the Twitter body; it is always posted separately as a comment/reply by the notifier. `test-post.ts` powers the admin "Send test post" button by reusing the same notifier with a synthetic payload. `cli-helpers.ts` is the shared OAuth-bootstrap CLI used by `scripts/social-*.ts`. **Failed-post Slack alert:** `workers/linkedin-post.ts` and `workers/twitter-post.ts` each call `slackNotifier.notifyPublishFailed({ runId, channel })` when the notifier returns `status: "failed"`. The alert is idempotent via `notification_state.linkedinFailure` / `twitterFailure`; a throwing webhook call logs `slack.publish_failed.failed` but the marker is NOT written so a retry can re-alert; the worker never throws on Slack failures.

## Rules
- No HTTP framework â€” this is a standalone Node process
- Workers call plain collector/processor functions â€” no business logic in workers
- Use repository factories (e.g. `createRawItemsRepo(db)`) for DB access â€” value imports of `@newsletter/shared/db` and `drizzle-orm` are only allowed inside `src/repositories/**` (enforced by `newsletter/enforce-repository-access`)
- Jobs must be idempotent â€” safe to retry
- Use `@pipeline/*` path aliases, never relative imports
- `ANTHROPIC_API_KEY` is validated at worker startup (not per job) â€” stage-2 rerank always needs Anthropic. `RANKING_MODEL` defaults to `claude-haiku-4-5-20251001`.
- **Archive-level idempotency markers are broadcast-only.** The fields `run_archives.email_sent_at`, `run_archives.linkedin_posted_at`, `run_archives.twitter_posted_at`, and every key in `run_archives.notification_state` JSONB are written ONLY from the canonical scheduled/broadcast path. Any worker that supports a targeted/per-recipient/manual variant (e.g. the welcome back-issue email sent via `subscriberIds: [<id>]`) MUST short-circuit before stamping these markers and before firing the corresponding Slack summary. Per-recipient dedup belongs on a per-recipient table (e.g. `email_sends`). This convention prevents the class of bug where a targeted send silently poisons the next broadcast (see commit 60d748b for the canonical fix in `email-send.ts`).
